1. How the service calls AI Studio

The app authenticates with the workspace ID + API key to obtain a JWT, then issues one HTTPS POST per LLM task:

# Auth (cached JWT, auto-refresh)
POST {AISTUDIO_BASE_URL}/auth        # {ID: workspaceId, Password: apiKey} -> JWT

# Each task call
POST {AISTUDIO_BASE_URL}/genai/{API_CODE}/prompt/{PROMPT_INDEX}
Authorization: Bearer {JWT}
Body: {
  "empNo": "...",
  "historyId": null,
  "parameters": [
    { "key": "SYSTEM", "value": "<persona / instructions>" },
    { "key": "USER",   "value": "<task content>" }
  ]
}
# Response: generated text (+ token usage: inputTokens / outputTokens)

The persona and all task content are supplied by the app at call time in the SYSTEM and USER parameters. The registered prompt only needs to pass those two parameters through to the model — so a single registered prompt can serve many tasks.

2. What we need from the platform: two pass-through prompts

To run the dual-model design (heavy generation on a strong model, frequent one-liners on a light model), please register two prompts, each just forwarding {SYSTEM} + {USER}:

PromptBind to modelUsed forOutput sizeCall share
PROMPT AClaude Sonnet 4.6Chapter 2/3 summaries & result-page (결과지) generation — quality-criticallarge (~600–2,200 out tokens)~20–25% of calls
PROMPT BGemini 2.5 FlashL-OWL line-by-line reflections during the flow — light & frequentsmall (~100–1,000 out tokens)~75–80% of calls
Why split: the light one-liners are the bulk of the volume. Routing them to Gemini frees the strong model for the few heavy summaries, lowering latency & cost and easing concurrency pressure.

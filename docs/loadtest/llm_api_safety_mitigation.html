AI Studio API Call Safety & Concurrency Mitigation

LG Magazine (mybook.lgacademy.com) — LLM request resilience under event load

Prepared 2026-06-24 · Build main a7e00c0 · Provider: LG AI Studio (aistudio.singlex.com)

1. Executive Summary

The magazine's personalized content is generated by LLM calls routed through LG AI Studio. The single hard constraint is concurrency: the AI Studio account returns generic HTTP 500 errors once roughly ~20 simultaneous in-flight calls are exceeded. This is an account / gateway limit — it is not solved by the 10-key rotation (which only spreads token quota) and is not changed by which model version is used.

Token quota is NOT the bottleneck. LG's stated per-minute limits are very large (Claude Sonnet 4.6 = 6,000,000 tokens/min). At our concurrency ceiling we use only a small fraction of that. The problem to mitigate is concurrent calls, not tokens.

Mitigation is in two parts: (A) an application-level safety stack that is already shipped in the current build, and (B) operational actions — most importantly asking LG to raise the concurrency limit, since that is the only lever that increases real capacity.

4. Configuration Rules (must hold in production)

  1. Keep concurrency under the account ceiling: AISTUDIO_MAX_CONCURRENCY × replicas < limit. Current: 5 × 3 = 15 < ~20 ✓. Adding API codes does not raise this (the wall is per-account, not per-code).
  2. Async needs sticky sessions. The job queue is in-memory per replica, so enable LLM_ASYNC=1 together with the nginx upstream change hash $cookie_qrius_session consistent; — otherwise a poll can land on a replica that doesn't hold the job (404).
  3. Timeouts must align. In sync mode keep AISTUDIO_QUEUE_TIMEOUT_MS (20s) under nginx proxy_read_timeout (≥60s). In async mode requests return instantly, so this is moot.

6. On "switch to Sonnet 4.6 to handle 100+ concurrent"

Model version ≠ concurrency capacity. Switching 4.5 → 4.6 will not, by itself, unlock 100 concurrent — the ~20 ceiling is an account property, not a model property. It may help only if LG confirms Sonnet 4.6 carries a higher concurrency allowance (they have not stated one).
  • The model swap is a dashboard rebinding (LG / workspace owner), not a code change; our only lever is which codes/prompt index we point to.
  • To reach 100 concurrent you need LG to raise the account concurrency limit. A concurrency-sweep test (10→20→40→60→100) can measure today's ceiling but cannot raise it, costs tokens, and should be coordinated with LG and run off-peak on a test code.

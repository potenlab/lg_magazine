1. Ringkasan Kesimpulan

Konfigurasi saat ini mampu menangani 100 pengguna bersamaan secara stabil. Sebelumnya (mode sinkron), pada 50 pengguna keberhasilan 41.8% dan pada 600 pengguna anjlok ke 6.5%. Setelah antrian asinkron + gerbang konkurensi, 100 permintaan bersamaan mencatat keberhasilan terima dan hasil akhir sama-sama 100.0%. Kegagalan diserap menjadi penundaan singkat, bukan error.
  • Keberhasilan akhir 100.0%: 100 permintaan mencapai hasil — 0 fallback/timeout/error hulu.
  • Penerimaan cepat: POST langsung mengembalikan 202 + jobId (p95 479ms). Pengguna tidak menunggu, layar tetap berjalan.
  • Penundaan hasil: waktu ke hasil median 3.6 detik, p95 4.7 detik, maks 10.8 detik — termasuk antre gerbang.
  • Gerbang konkurensi: 5 panggilan serentak/proses × 3 replika ≈ 15, dijaga di bawah batas AI Studio hulu (~20).
  • Model: diukur pada Claude Sonnet 4.6.

4. Dibanding Sebelumnya (Dampak Perbaikan)

Pada mode sinkron lama, panggilan LLM diproses langsung saat permintaan sehingga saat beban bersamaan menumpuk, AI Studio hulu mengembalikan error 500 per panggilan (50 pengguna 41.8%, 600 pengguna 6.5% sukses). Antrian asinkron + gerbang konkurensi membatasi jumlah panggilan serentak di bawah batas hulu dan mengantre kelebihannya sebentar, mengubah error menjadi penundaan. Hasilnya, pada 100 pengguna bersamaan keberhasilan akhir mencapai 100.0%.

5. Rekomendasi

  • Pertahankan konfigurasi: kelas berisi 100 orang stabil dengan setelan saat ini. Tak perlu tindakan tambahan.
  • Siapkan skala lebih besar: di atas 600 pengguna penundaan hasil bisa memanjang — sesuaikan jumlah replika atau nilai gerbang (× replika ≤ ~20) terhadap batas hulu.
  • UX indikator progres: karena terima instan (202), gunakan indikator "sedang membuat…" agar tunggu terasa wajar.
  • Pemantauan: lacak p95 waktu ke hasil dan panjang antrean sebagai metrik operasi untuk scale-up dini.
  • Uji skala besar: disarankan uji tambahan skenario 300·600 pengguna bersamaan pada dini hari untuk memetakan kurva penundaan.

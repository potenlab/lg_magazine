1. Ringkasan Kesimpulan

Konfigurasi saat ini tidak menggagalkan satu pun permintaan, bahkan pada 600 panggilan bersamaan. Antrian asinkron + gerbang konkurensi menyerap lonjakan sesaat menjadi penundaan, bukan error. Penerimaan instan (202, p95 318ms); hasil tiba lewat antrean pada median 14.9 detik · p95 42.2 detik · maks 48.5 detik. Server produksi normal sepanjang uji.
  • Keberhasilan akhir 100.0%: seluruh 600 permintaan mencapai hasil — 0 error hulu, 0 polling 404, 0 timeout klien.
  • Penerimaan cepat: POST langsung mengembalikan 202 + jobId (p95 318ms). Pengguna tak menunggu, layar berjalan.
  • Penundaan hasil (harga yang dibayar): makin tinggi konkurensi, makin lama antre — waktu ke hasil median 14.9 detik, p95 42.2 detik.
  • Gerbang konkurensi: 5/proses × 3 replika ≈ 15, dijaga di bawah batas AI Studio (~20) → error 500 hulu dicegah dari akar.
  • Model: diukur pada Claude Sonnet 4.6.

5. Interpretasi — Arti Hasil Ini

Intinya: "600 pengguna bersamaan pun tidak membuatnya rusak." Pada mode sinkron lama, panggilan bersamaan langsung memicu error 500 AI Studio hulu; antrian asinkron kini membatasi jumlah panggilan serentak di bawah batas dan mengantre kelebihannya hingga semua berhasil. Harganya adalah penundaan hasil (pada 600: median 14.9 detik, p95 42.2 detik) — masih wajar diserap dengan indikator "sedang membuat…".

6. Rekomendasi

  • Pertahankan konfigurasi: hingga 600 panggilan bersamaan tanpa gagal. Bisa dioperasikan tanpa tindakan tambahan.
  • Bila perlu mempercepat: naikkan jumlah replika atau nilai gerbang (× replika ≤ ~20) terhadap batas hulu → penundaan hasil berkurang.
  • UX indikator progres: karena terima instan (202), gunakan "sedang membuat…" untuk meringankan tunggu p95 42.2 detik.
  • Pemantauan: lacak p95 waktu ke hasil dan panjang antrean sebagai metrik operasi untuk scale-up dini.

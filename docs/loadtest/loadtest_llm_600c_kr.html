1. 결론 요약

현재 구성은 600명 동시 호출에서도 한 건도 실패하지 않았습니다. 비동기 큐 + 동시성 게이트가 순간 폭주를 오류 대신 대기로 흡수했습니다. 접수는 즉시(202, p95 318ms) 이뤄지고, 결과는 큐를 거쳐 중앙값 14.9초·p95 42.2초·최대 48.5초에 도착했습니다. 운영 서버는 전 구간 정상이었습니다.
  • 최종 성공률 100.0%: 600건 요청 전부 결과까지 도달 — 상위 오류 0, 폴링 404 0, 클라이언트 타임아웃 0.
  • 빠른 접수: POST는 즉시 202 + jobId 반환 (p95 318ms). 사용자는 기다리지 않고 화면 진행.
  • 결과 지연(대가): 동시성이 클수록 큐 대기가 늘어 결과까지 중앙값 14.9초, p95 42.2초.
  • 동시성 게이트: 프로세스당 5건 × 3 레플리카 ≈ 15건으로 상위 AI Studio 한계(~20)를 넘지 않게 직렬화 → 상위 500 오류 원천 차단.
  • 모델: Claude Sonnet 4.6 기준 측정.

4. 100명 대비 (확장성)

같은 방식의 100명 테스트와 비교하면, 동시성이 6배로 늘어도 성공률은 100%로 동일하고 결과 지연만 증가합니다. 즉 현재 로직은 실패하지 않고 대기로 부하를 흡수합니다.

5. 해석 — 이 결과의 의미

"600명이 동시에 써도 깨지지 않는다"가 핵심입니다. 이전 동기 방식에서는 동시 호출이 상위 AI Studio 500 오류로 직결됐지만, 현재 비동기 큐는 동시 호출 수를 한계 이하로 직렬화하고 초과분을 대기시켜 전부 성공으로 끝냅니다. 비용은 결과 지연(600명에서 중앙 14.9초, p95 42.2초)이며, "생성 중…" 진행 표시로 자연스럽게 흡수 가능한 수준입니다.

6. 권장 사항

  • 현 구성 유지: 600명 동시 호출까지 무실패. 추가 조치 없이 운영 가능.
  • 지연 단축이 필요하면: 레플리카 수 또는 게이트 값(× 레플리카 ≤ ~20)을 상위 한계에 맞춰 상향 → 결과 지연 감소.
  • 진행 표시 UX: 접수가 즉시(202)이므로 "생성 중…" 표시로 p95 42.2초 대기를 체감 완화.
  • 모니터링: 결과까지 p95와 큐 길이를 운영 지표로 추적해 한계 도달 전 선제 확장.
측정 한계. 단일 작업(extractKeyword, 1건/사용자) 기준의 순수 동시성 측정입니다. 실제 한 사용자는 한 세션에서 ~21개 작업(무거운 합성 포함)을 순차 호출하므로, 전체 매거진 완주 관점의 대규모(예: 600 세션) 측정은 결과 지연이 훨씬 길어집니다. 본 보고서는 "현재 로직이 600 동시 호출을 실패 없이 처리하는가"에 대한 답입니다(=예).

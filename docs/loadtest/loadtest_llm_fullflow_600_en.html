1. Executive Summary

At 600 concurrent full sessions, task success is 90.5% and session-complete is 0.0%. Splitting light tasks onto the Gemini 2.0 Flash lane relieved the heavy Claude Sonnet 4.5 lane, but 7 heavy tasks per session accumulate and some did not return within the 5-minute deadline. Per-task success is high; session-complete (all 21 tasks succeeding) is the stricter bar.
  • Dual-model routing: 14 light tasks → Gemini 2.0 Flash (dedicated lane), 7 heavy → Claude Sonnet 4.5. Cuts the bottleneck lane from 21 to 7 calls per session.
  • Task success 90.5%: across 600 sessions × 21 tasks ≈ 21 tasks.

5. Interpretation — What This Means

Splitting light tasks onto Gemini 2.0 Flash cleared the full saturation seen before dual-model. But at 600 full sessions the 7 Claude Sonnet 4.5 tasks per session accumulate, and some don't finish within the deadline. The heavy lane's throughput (replicas × gate) is the next bottleneck.

6. Recommendations

  • Scale the heavy lane: raise replicas or the gate value (× replicas ≤ upstream cap) to increase Claude Sonnet 4.5 throughput.
  • Revisit the deadline: check whether the 300s per-task deadline is sufficient vs heavy p95 (26.8s).
  • Progress UX: absorb the wait with a "generating…" indicator.
Measurement Caveat. This is the heaviest scenario: 1 users running the full magazine flow (21 tasks each) concurrently. Real users have longer read/type gaps between tasks that spread the load, so this test is a conservative (near-worst-case) upper bound. Dual-model routing (light → Gemini 2.0 Flash, heavy → Claude Sonnet 4.5) was applied exactly as the production default.

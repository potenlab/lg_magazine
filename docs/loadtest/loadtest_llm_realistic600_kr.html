1. 결론 요약

수정은 옳았고 효과가 있었습니다. 경량 레인 코드를 1개(GEMINI_FLASH)에서 10개(LG_BOOK_GEMINI 외 9개)로 늘리자 "코드 소진" 오류가 77% 줄고 작업 성공률이 28.7%→45.8%로 올랐습니다. 배포 전 10개 코드 모두 10/10 정상 확인, 20개 동시 버스트도 20/20 성공했습니다. 그러나 600명이 동시에 풀세션(21작업, 총 ≈12,600작업)을 돌리면, 이제 병목은 코드 수가 아니라 동시 처리량(게이트 ~15 + AI Studio 계정 ~20 동시 한계)입니다. 작업이 오류 대신 큐에서 대기하다 300초 마감을 넘겨 타임아웃됩니다(라이브 프로브: 큐 위치 188~337).
병목 이동 — 코드 소진 → 동시성 처리량. 수정 전: 경량 코드 1개가 즉시 소진되어 오류 발생. 수정 후: 코드는 충분하지만 동시 실행 슬롯(게이트 ~15, 계정 ~20)이 600 동시 부하를 못 따라가 큐 적체→타임아웃. 즉 코드 증설은 쿼터/소진 문제를 풀지만 동시 처리량은 늘리지 못합니다 — 처리량은 게이트와 계정 한계가 결정합니다.
  • 작업 성공률 28.7% → 45.8% (상대 +60%): 명확한 개선.
  • 상위 오류 4,156 → 963건 (−77%): 소진 오류 대부분 제거.
  • 실패 양상 전환: 타임아웃 170→1,055건 — 오류로 죽는 대신 큐에서 대기하다 마감 초과.
  • 경량 TTR 93.5→236.0초(중앙값): 빨리 실패하던 게 사라지고 깊은 큐에서 대기 → 수치상 증가.
  • 무거운(Sonnet) TTR: 중앙 296.6초, p95 302.0초 — 무거운 레인도 동시성 한계로 포화.
  • 완주 세션 0/600: 600 동시 풀세션은 현재 동시 처리량 천장을 초과.

4. 해석 — 이 결과의 의미

코드 증설은 정확한 1차 수정이었습니다(소진 오류 제거). 그러나 600명 동시 풀세션은 ≈12,600개 작업을 동시에 밀어 넣는 극단적 부하로, LG AI Studio 계정의 ~20 동시 한계와 로컬 게이트(~15)가 처리량 천장입니다. 실사용은 600명이 정확히 같은 순간 시작하지 않고 자연스럽게 분산되므로, 실제 체감 부하는 이 테스트보다 훨씬 완만합니다. 비동기 큐 덕분에 하드 실패 대신 대기로 흡수되는 점은 그대로 유효합니다.

5. 권장 사항

  • 완료 (검증됨): 경량 코드 1→10개 — 소진 오류 77% 감소. 운영 반영 완료.
  • 동시성 한계 확인(다음 단계): LG AI Studio 계정 ~20 동시 제한이 두 레인 합산인지 레인별인지 확인. 상향 가능하면 협의.

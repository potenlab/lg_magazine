## 2. What user action causes the bandwidth spike

A "cold visit" is the worst case. The first time a participant opens
`mybook.lgacademy.com` (no browser cache yet), the browser fetches in parallel:

| Asset class | Count per cold visit | Approx bytes | Source path |
|---|---:|---:|---|
| Next.js JS/CSS chunks (content-hashed) | ~9 files | ~1.5 MB | `/_next/static/*` |
| Background images (webp) | 4–8 files | 1–3 MB | `/vision_express/common/*.webp` |
| Audio (mp3 — BGM + SFX) | 1–3 files | 2–6 MB | `/vision_express/*.mp3` |
| Fonts (woff2 / ttf) | 2–4 files | 1–2 MB | `/fonts/*` |
| Brand SVG | 1 file | ~30 KB | `/brand/*.svg` |
| **Per-cold-visit total** | **~20 requests** | **~5–10 MB** | |

Once a user has visited, the browser caches these files. **A returning visitor
costs near-zero bytes** for the same assets (provided cache headers are correct
— see §5.1). The bottleneck is therefore concentrated in the **first 30–60
seconds** of a training event, when many fresh browsers cold-load at once.

## 4. Failure mode — what actually happens when we hit the wall

What an external observer (LG IT monitoring) would see during a saturation
event:

| Layer | Symptom |
|---|---|
| **lg_magazine end users** | Blank page / partial render. JS chunks 60-second timeout, app does not start. Browser shows broken images and missing fonts. |
| **lg_magazine origin VM** | nginx 200 OK with high `$bytes_sent`/`$request_time`; Node CPU normal — the bytes are leaving slowly, not the requests slow to compute. |
| **DMZ shared pipe** | Sustained 100–120 Mbps from one tenant. TCP retransmits rise. Queue depth at the DMZ uplink grows. |
| **Other tenants (Qrius etc.)** | Increased TCP latency on their own egress, even though their VMs are idle. **This is the blast-radius concern.** |

**The failure is invisible at the VM level.** lg_magazine's own server metrics
look fine (CPU OK, memory OK, nginx returns 200s). The bottleneck is in the
*pipe between the VM and the internet*, which we cannot directly observe.

## 5. Mitigation plan — fitting inside the envelope

The four mitigations below, applied together, are projected to reduce the
**1,000-user egress demand from ~120 Mbps to ~20–30 Mbps** — staying well
inside any plausible fair share of the 500 Mbps pipe.

### 5.1 Browser-cache the static assets (immutable, 1-year)

Set `Cache-Control: public, max-age=31536000, immutable` on `_next/static/*` and
`/fonts/*`; `max-age=2592000` (30d) on `/vision_express/*`.

- **Effect:** repeat visits download **0 bytes** of static assets.
- **Real-world impact at LG 인화원:** training participants typically open the
  app multiple times during a session. After the first cold load, every
  subsequent open costs only the small dynamic HTML + API JSON (~50 KB).
- **Implementation:** [production_deployment.md §3](production_deployment.md) — no
  app code change; only an nginx configuration block on the same VM.

### 5.2 Compress JS/CSS/SVG on the wire (gzip + brotli)

Pre-compress all text assets at build time and serve `.gz` / `.br` siblings
through nginx.

- **Effect:** ~70% reduction in JS/CSS bytes leaving the pipe.
- Per-cold-visit drop: ~1.5 MB → ~0.5 MB for the Next.js bundle.
- **Implementation:** see [production_deployment.md §3c](production_deployment.md) (gzip/brotli tuning).

### 5.3 Asset diet — re-encode the heavy media

| Asset | Now | Proposed | Why safe |
|---|---:|---:|---|
| `kokoreli777-inside-old-train-169418.mp3` | 3.7 MB | ~1 MB (96 kbps mono) | Ambient narration, no stereo needed |
| `freesound_community-writing-with-pen-35109.mp3` | 1.9 MB | ~500 KB | SFX, mono is fine |
| `Chapter_01-2.webp` | 1.1 MB | ~400 KB (AVIF) | Same visual fidelity at ~3× smaller |
| `l-owl-09.png` (and siblings) | 1.0 MB | ~250 KB (lossless webp) | PNG → webp lossless |

- **Effect:** per-cold-visit total drops from 5–10 MB → 2–4 MB.
- **Implementation:** offline batch re-encode; one PR.

### 5.4 Lazy-load chapter assets

Audit the React tree and confirm chapter-N media only fetches when the user
reaches chapter N (and not on the landing page).

- **Effect:** spreads bandwidth demand across the full session duration
  (~30 min) instead of concentrating it in the first 30 seconds.
- **Implementation:** code audit + targeted dynamic import. Half a day.

### Combined projection

| Layer applied | 1,000-user egress | Share of 500 Mbps pipe |
|---|---:|---:|
| Today (no mitigation) | ~120 Mbps | 24 % |
| + 5.1 (immutable cache) | ~60 Mbps (cold-load only) | 12 % |
| + 5.2 (compression) | ~35 Mbps | 7 % |
| + 5.3 (asset diet) | ~25 Mbps | 5 % |
| + 5.4 (lazy load, spread over session) | **~10–15 Mbps sustained** | **~2–3 %** |

**At ~10–15 Mbps sustained, lg_magazine becomes a small co-tenant** — well
inside polite use of the shared pipe.

## 6. Requests to LG IT

To close the gap from "projected ~10–15 Mbps" to "verified safe at training
day", two items need LG IT support. Neither requires network expansion.

### 6.1 Domestic CDN approval (high impact)

May we put `/_next/static/*`, `/vision_express/*`, `/brand/*`, `/fonts/*`
behind a **Korean-domestic CDN** (candidates in preference order: **LG U+ CDN**,
KT Cloud CDN, Naver Cloud CDN, Akamai KR)?

- **Effect:** static bytes are served from the CDN edge, **not from LG's
  DMZ pipe at all**. The lg_magazine VM only ships HTML + API JSON (~50 KB
  per user), reducing peak egress demand by **~95%**.
- **Why ask LG IT:** putting `mybook.lgacademy.com` behind a CDN involves a
  DNS / TLS-cert decision and a security review. We expect the LG U+ option to
  be lowest-friction because it is the same carrier family.
- **Fallback if no CDN is approved:** §5.1–5.4 still apply and are sufficient
  for a single ~1,000-user training event provided §6.2 is also in place.

### 6.2 Bandwidth telemetry visibility (low effort)

Can LG IT share a graph of **DMZ pipe utilization** during the next scheduled
training event? Even a daily peak-Mbps line for the 500 Mbps pipe lets us
verify our envelope projection against reality and adjust before the next
training.

Without this, we are projecting from our own egress data only and cannot see
co-tenant interaction.

## 7. Operational ask (인화원 운영팀)

If the training schedule typically opens `mybook.lgacademy.com` to all
participants at a single instant (e.g. 09:00 정각), peak demand will arrive in
a 5–10 second window. **Spreading the start over 5–10 minutes** (staggered
cohort entry, or sending the link 5 min before the official start) reduces
peak Mbps by ~10× with zero technical work.

This is a request to operations, not engineering. We mention it because it is
the single highest-leverage mitigation available.

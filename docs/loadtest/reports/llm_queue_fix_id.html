1. Ringkasan Kesimpulan

Antrian asinkron + gate konkurensi + nginx sticky session di-deploy ke produksi, lalu beban 600 pengguna diukur ulang. Kegagalan akibat overload (89% → 0%) hilang sepenuhnya; semua pengguna diterima (202) lalu diproses bergiliran. Routing sticky tetap 0 error pada beban 600. Sisanya bukan soal 'gagal' melainkan 'kecepatan' — waktu tunggu akibat batas throughput.

2. Apa yang Diperbaiki

  • Antrian asinkron: POST langsung mengembalikan jobId (202), klien polling hasil. Koneksi tidak ditahan, jadi tidak ada timeout — pengguna mengantre.
  • Gate konkurensi: panggilan hulu dibatasi 5 per replika (×3=15), sisanya menunggu. Mencegah 500 overload dari sumbernya.
  • nginx sticky session: hash $cookie_qrius_session consistent — tiap pengguna dipaku ke satu replika agar polling selalu menemukan job-nya.

4. Waktu Respons (harga sebuah antrian)

Sebelum: hasil diterima6.5%
Sesudah: hasil diterima90.2%
Antrian mengubah 'gagal' menjadi 'menunggu'. 600 pengguna melewati pipa berkapasitas 15 konkuren, sehingga waktu ke hasil: median 2.5 mnt, p95 4.7 mnt, maks 5.0 mnt. Saat batas pengukuran (5 mnt), 55 pengguna masih mengantre (bukan gagal — menunggu giliran; aplikasi nyata menunggu hingga 6 mnt).

5. Pekerjaan Tersisa (kecepatan)

  • Naikkan batas hulu (utama): naikkan batas konkurensi AI Studio di atas ~20 atau gunakan akun terpisah — satu-satunya solusi akar untuk mengurangi waktu tunggu.
  • Sebar kedatangan: masukkan 600 pengguna bertahap dalam beberapa menit agar antrian sesaat jauh berkurang.
  • Naikkan konkurensi antrian: LLM_QUEUE_CONCURRENCY bisa dinaikkan tapi hanya efektif dalam batas hulu ~20 (6/replika → 18).
  • Catatan: ini tugas paling ringan — majalah nyata (21 sintesis berat) menunggu lebih lama. Langkah di atas tetap diperlukan.

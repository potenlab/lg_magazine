1. 이 스트레스 테스트에 대하여

스트레스(부하) 테스트는 많은 사람이 한꺼번에 사용할 때 앱이 빠른 속도를 유지하는지, 그리고 안정적인지를 측정합니다. 실제 600명을 동시에 접속시키는 대신, 널리 쓰이는 오픈소스 부하 테스트 도구 k6로 수백 명의 동시 사용자가 실제 HTTP 요청을 운영 사이트에 보내는 상황을 시뮬레이션하고, 각 가상 사용자의 체감 값을 기록했습니다.

각 가상 사용자는 실제 독자처럼 매거진을 엽니다. 페이지·이미지·폰트·인트로 오디오를 받는 첫 "콜드 방문" 후, 캐시된 상태의 빠른 "웜" 경로로 챕터를 이동하며, 실제 사람이 읽듯 동작 사이에 2~5초씩 멈춥니다.

점진적 입장 모델을 사용했습니다 — 모두가 같은 순간이 아니라 짧은 시간에 걸쳐 입장하여 실제 수업이 차오르는 방식을 모사합니다. 이를 운영 서버(라이브)에 대해 6단계(동시 100·200·300·400·500·600명)로 수행하고, 각 단계의 성공률·응답시간(p95)·처리량을 측정했습니다.

3. 표 읽는 법

성공률 — 약 100% 유지
600명에서도 첫 시도 실패는 0.17%에 불과. 사실상 오류 없음.
홈 · 앱 내 이동(p95) — 평탄
가장 중요한 값, 즉 서버 자체 응답 시간입니다. 100명에서 600명까지 거의 변하지 않습니다(약 130ms, 265ms). 부하가 늘어도 느려지지 않는다는 뜻이며, 여유가 충분합니다.
첫 방문(p95) — 유일하게 증가하는 항목
처음 매거진을 열 때 브라우저가 이미지·폰트·인트로 오디오를 받습니다. 이 다운로드가 가장 무거운 순간이며 대역폭에 좌우됩니다. 사용자당 1회성 비용이고, 이후 모든 페이지는 빠른 "웜" 경로입니다. (이 값은 단일 테스트 PC의 인터넷 회선에도 일부 영향을 받으므로, 실제 서버에서의 첫 방문 경험은 표시값보다 나쁘지 않습니다.)
처리량 — 매끄럽게 증가
접속자 수에 비례해 증가(58 → 147.3 Mbps), 가용 네트워크 용량 내에서 여유.

4. 의미

동시 600명은 여유 있게 처리되며, 추가 여유도 충분합니다. 전 구간에서 응답 시간이 평탄하고 오류는 사실상 0%입니다. 목요일 수업을 위해 서빙 계층에서 바꿀 것은 없습니다.
관리할 것은 첫 방문 다운로드 집중뿐입니다. 600명이 정확히 같은 몇 초에 매거진을 열면 무거운 첫 로딩이 한꺼번에 발생합니다. 입장을 1~2분에 걸쳐 분산하면 매끄럽게 유지됩니다. 이미 적용한 자산 경량화도 이 비용을 줄여줍니다.

5. 목요일(600명) 권장 사항

  • 입장 분산: 가능하면 1~2분에 걸쳐 입장 — 동시 첫 로딩 완화.
  • 모니터링: 수업 시작 시각에 정상 여부 확인.
  • 용량 변경 불필요: 위 수치는 운영 서버 실측이며 이미 통과.

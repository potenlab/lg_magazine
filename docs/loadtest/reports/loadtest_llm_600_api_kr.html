1. 결론 요약

600명 동시 버스트에서 접수 100.0%, 최종 성공 100.0%. 비동기 큐가 순간 폭주를 오류 대신 대기로 흡수하고, 듀얼 모델 도입 후 가벼운 작업은 Gemini 2.0 Flash (light lane) 전용 라인에서 처리됩니다. 6월 단일 레인(Sonnet) 측정의 중앙값 14.9초 대비 이번 중앙값은 6.9초입니다.
  • 최종 성공률 100.0%: 600건 요청 기준 — 폴링 404 0건 · 상위 오류 0건 · 타임아웃 0건.
  • 빠른 접수: POST는 즉시 202 + jobId 반환 (p95 463ms). 사용자는 기다리지 않고 화면이 진행됨.
  • 결과 지연: 중앙값 6.9초 · p95 32.4초 · 최대 259.6초 — 큐 대기 포함.
  • 동시성 게이트: 프로세스당 5건 × 3 레플리카 ≈ 15건으로 상위 한계 이하 직렬화.
  • 모델: Gemini 2.0 Flash (light lane) 기준 측정 (듀얼 레인 운영 기본값).

4. 이전 측정과 비교

초기 동기 방식은 600명 동시에서 성공률 6.5%로 붕괴했고, 비동기 큐 도입(6월 25일, 단일 레인 Sonnet) 후 100% 성공·중앙값 14.9초·p95 42.2초를 기록했습니다. 이번 측정(듀얼 레인)에서는 가벼운 작업이 Gemini 2.0 Flash (light lane) 라인으로 분리되어 성공률 100.0% · 중앙값 6.9초 · p95 32.4초입니다.

5. 권장 사항

  • 현 구성 유지: 600명 동시 버스트를 현재 설정이 감당합니다.
  • 진행 표시 UX: 접수가 즉시(202)이므로 "생성 중…" 진행 표시로 체감 대기를 자연스럽게.
  • 모니터링: 결과까지 p95와 큐 대기 길이를 운영 지표로 추적.
  • 풀 플로우 참고: 전체 세션(23개 작업) 부하는 별도 풀 플로우 테스트로 측정 — 부분 실측(11분)에서 작업 성공률 99.4% 확인.
측정 한계. 단일 가벼운 작업(extractKeyword) 기준 1회 실행 — 무거운 합성(Sonnet 라인) 작업의 지연은 본 테스트 범위 밖입니다. 큐는 프로세스(레플리카) 내 메모리 기반이므로 접수와 폴링이 같은 레플리카로 라우팅되어야 하며(nginx 세션 해시), 폴링 404=0건으로 라우팅 상태를 확인했습니다.
